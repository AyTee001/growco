# Робота з репозиторієм

Щоб уникнути конфліктів у коді та зберегти стабільність проєкту, у цьому репозиторії діють наступні правила.

---

## 🔀 Робота тільки через Pull Request

Усі зміни вносяться до основної гілки master **виключно через Pull Request**.

Типовий процес роботи:

1. Створіть нову гілку від `master`
2. Виконайте зміни у своєму коді
3. Зробіть `push` у свою гілку
4. Створіть **Pull Request**
5. Після перевірки зміни будуть об'єднані з `master`

## 🌿 Назви гілок

Назва гілки повинна коротко описувати зміни і зазначати ідентифікатор завдання (якщо є пов'язане завдання)

## 🚫 Заборонено пушити напряму в `master`

Усі зміни повинні проходити через **Pull Request**.

## 🛠 Проблеми з доступом

Якщо у вас:

- немає доступу до репозиторію
- неможливо створити гілку
- не виходить запушити зміни
- виникли проблеми з Pull Request чи треба його перевірити

зверніться до **відповідальних за репозиторій**

## 🔌 API Service Proxy (Hey-API)

Ми використовуємо @hey-api/openapi-ts для генерації типізованого SDK на основі специфікації NestJS OpenAPI (Swagger). Для коректного відтворення типів необзідно виконати додаткові налаштування. На цей час використовуються атрибути на властивостях сутностей. Їх приклад можна побачити у файлі /src/entities/Products.ts.

1. Генерація клієнта
  Кожного разу, коли змінюється API на бекенді (нові маршрути, змінені DTO), треба виконати цю команду в кореневій папці фронтенду:

  ```bash
    npm run generate-api
  ```
  При цьому бек має бути запущений!
  
2. Для виклику ендпоінтів просто імпортуємо згенеровану функцію.

  ``` bash
    const { data, error } = await productsControllerFindAllByCategory({
      path: { categoryId: id }
    })
  ```

## 🏗️ Налаштування бази даних (PostgreSQL)

Проєкт використовує PostgreSQL та TypeORM. Конфігурація БД зберагіється у файлі /src/config/typeorm.ts, де можна поправити порт та креди до БД, якщо треба. Щоб запустити базу даних та синхронізувати схему, виконай наступні кроки.

1. Вимоги

  - Встановити PostgreSQL або використати Docker.
  - Створити базу даних з назвою growco_db.

2. Міграції TypeORM

  Ми використовуємо міграції для керування схемою бази даних. Не створюйте таблиці вручну.

  - Після кожної зміни у властивостях сутності, що впливає на БД, ОБОВ'ЯЗКОВО після застосування усіх доступних міграцій, генеруємо нову

    ```Bash
      npx typeorm-ts-node-commonjs migration:generate src/migrations/<ім'я міграції> -d src/config/typeorm.ts
    ```

  - Щоб застосувати усі поточні міграції до БД треба виконати наступну команду

    ```Bash
      npx typeorm-ts-node-commonjs migration:run -d src/config/typeorm.ts
    ```

  - Щоб відкотити останню міграцію

    ```Bash
      npx typeorm-ts-node-commonjs migration:revert -d src/config/typeorm.ts
    ```

3. Seeding (після застосування усіх міграцій; виконувати скрипти з кореневої папки проєкту). Команди необхідно виконувати ВИНЯТКОВО У ЦЬОМУ ПОРЯДКУ

   - npm run seed:categories
   - npm run seed:products
